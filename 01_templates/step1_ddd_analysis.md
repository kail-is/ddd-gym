# DDD 분석 템플릿
요구사항을 해석해 설계 후보를 만들고, 선택/트레이드오프를 명시합니다.

## 체크리스트
- [ ] 컨텍스트/애그리게잇/이벤트 후보가 각각 1개 이상이다.
- [ ] 규칙을 정책 vs 불변조건으로 분류했다.
- [ ] 트랜잭션 경계를 설명했다.
- [ ] 선택한 방식과 버린 대안을 비교했다.
- [ ] 트레이드오프를 명시했다.

## 문서 구조
1. **Bounded Context 후보**
   - 예시: 예약관리, 정산관리, 체크인관리
2. **Aggregate 후보**
   - 예시: 예약(루트: Booking), 정산(루트: Settlement)
3. **Domain Event 후보**
   - 예시: BookingConfirmed, BookingCancelled, CheckInRecorded
4. **엔티티 / Value Object 후보**
   - 엔티티: Booking, StayProgress 등과 책임.
   - VO: 날짜 범위, 인원 수, 정책 파라미터 등.
5. **규칙 분류**
   - 불변조건: 확정된 예약은 같은 숙소/날짜 조합을 중복 점유할 수 없다.
   - 정책: 체크인 전 취소 패널티는 지역별 규칙으로 달라진다.
6. **트랜잭션 경계**
   - 예시: 예약 생성 시 재고 점유까지 한 트랜잭션.
7. **선택한 방식 vs 버린 대안**
   - 선택: 예약과 정산을 분리된 컨텍스트로 운영.
   - 버린 대안: 단일 컨텍스트에서 정산까지 즉시 처리.
8. **트레이드오프**
   - 예시: 일관성 vs 확장성, 운영 복잡도 증대 등.

> TIP: AI에게는 "정답을 말하지 말고, 이 설계의 경계/책임 약점을 질문으로 드러내달라"고 요청하세요.
