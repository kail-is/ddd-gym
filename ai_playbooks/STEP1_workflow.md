# 전체 워크플로우
요구사항 작성 → DDD 해석 → 시나리오 작성 → 테스트 설계 → AI 질문 루프 → 설계 수정 → 요약 노트 작성 → 토론 준비 → 오프라인 토론

| 단계 | 목적 | 산출물 | AI 사용법 | 다음 단계 조건 |
| --- | --- | --- | --- | --- |
| 요구사항 작성 | 문제 공간을 명확히 정의 | `01_templates/step1_requirement_card.md` 기반 문서 | 열린 질문 찾기, 빠진 주체/규칙 확인 질문 요청 | 모든 요구사항 항목(목적~열린 질문) 작성 |
| DDD 해석 | 요구사항을 도메인 모델 후보로 전환 | `01_templates/step1_ddd_analysis.md` | 경계/트랜잭션/정책 분류 질문을 받음 (정답 금지) | 컨텍스트/애그리게잇/이벤트/규칙/경계/대안/트레이드오프 작성 |
| 시나리오 작성 | 사용자 관점 행동 검증 | `01_templates/step2_scenario_template.md` | 경계 조건, 누락된 Actor 질문 받음 | 최소 1개 정상 시나리오 + 예외 시나리오 작성 |
| 테스트 설계 | 시스템 검증 포인트 명세 | `01_templates/step2_test_template.md` | 테스트 분류별로 빠진 케이스 질문 받음 | 정상/예외/경계/위험 케이스 각 1개 이상 |
| 질문 루프 | 설계 약점 노출 | `ai_playbooks/STEP0_AI_START_PROMPT.md` 안내에 따라 질문을 받아 `participants/<이름>/step0_prepare/weekly_progress.md`에 기록 | 정답 금지, 질문만 생성하도록 요청 | 주차별 질문 로그 업데이트 |
| 설계 수정 | 질문에 기반해 설계를 재구성 | 변경 로그 | AI 질문 중 설계 수정을 촉발한 항목에 대해 원인/결정 기록 | 변경 사항이 명시된 후 요약 노트의 "AI 질문으로 수정된 내용" 섹션 채움 |
| 요약 노트 작성 | 비동기 기록 준비 | `participants/sample/step3_integration/pr_outline.md` | 요약 품질 확인 질문 사용 | 요약 섹션을 모두 채움 |
| 토론 준비 | 오프라인 토의 포인트 정리 | `01_templates/step3_review_questions.md` | 다음 미팅에서 다룰 질문/쟁점 선정 | 최소 2개의 토론 포인트 정리 |
| 오프라인 토론 | 후보 설계 비교 및 선택 | `study_guides/meeting_guide.md` 프로토콜 | 비교 대상 선정 질문 사용 | 토론 노트/선택안 기록 |

## 실행 팁
- 각 문서에는 체크리스트가 있으니, 다음 단계로 넘어가기 전에 반드시 체크하세요.
- 단계는 독립 실행 가능하도록 설계되어 있으니, 특정 단계만 수행해도 요약 노트를 남길 수 있습니다.
- 논의/검토는 매주 오프라인 세션에서 진행하며, 코멘트 대신 대화/토론으로 정리합니다.
- 시간이 부족하면 요구사항+분석까지만 제출하고, 다음에 이어서 시나리오/테스트 단계부터 진행합니다.

## AI 챗봇 연동
- 챗봇은 `ai_playbooks/STEP1_chat_workflow.md` 시나리오를 그대로 따라 주차별 질문을 던집니다.
- 각 세션 시작 시 `shared_topics/stepX_*.md`를 요약해 공통 목표/제약을 다시 상기시킵니다.
- 질문을 받으면 해당 템플릿(예: `01_templates/...`)에 답을 적고, 완료 후 `participants/<이름>/step0_prepare/weekly_progress.md`에 로그를 남기면 다음 세션에서도 이어서 질문할 수 있습니다.
