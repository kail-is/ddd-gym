# 질문 루프 로그

## 체크리스트
- [x] 질문 5개 이상 기록됨.
- [x] 각 질문에 대한 응답/변경 사항 명시됨.
- [x] 설계에 영향을 준 질문 표시됨.

## 질문 로그 포맷

### 질문 1
- **영역**: 경계
- **질문 내용**: 정산 보류 상태를 예약 컨텍스트가 직접 판단해야 하나요, 아니면 정산 컨텍스트에 물어봐야 하나요?
- **내 응답/판단**: 정산 컨텍스트가 정책의 주인이니까 조회만 해야 한다.
- **설계 변경 여부**: O
- **변경 내용**: Booking이 HostSettlement 상태를 직접 관리하지 않고, API 조회 방식으로 변경.

### 질문 2
- **영역**: 책임
- **질문 내용**: 체크인 가능 여부는 Booking이 알아야 하나요, StayProgress가 알아야 하나요?
- **내 응답/판단**: 체크인은 현장 상태니까 StayProgress가 책임져야 한다.
- **설계 변경 여부**: O
- **변경 내용**: StayProgress 애그리게잇을 분리하고 체크인/체크아웃 전담.

### 질문 3
- **영역**: 상태
- **질문 내용**: 체크인 없이 체크아웃 API가 호출되면 어떻게 되나요?
- **내 응답/판단**: 이건 위험 케이스다. 막아야 한다.
- **설계 변경 여부**: O
- **변경 내용**: StayProgress에 상태 전이 가드 추가, 감사 로그 기록.

### 질문 4
- **영역**: 트랜잭션
- **질문 내용**: 예약 생성과 재고 점유가 다른 트랜잭션이면 어떤 문제가 생기나요?
- **내 응답/판단**: 중복 예약 가능성. 한 트랜잭션으로 묶어야 한다.
- **설계 변경 여부**: O
- **변경 내용**: 재고 조회 + Booking 생성 + 상태 전이를 단일 트랜잭션으로.

### 질문 5
- **영역**: 이벤트
- **질문 내용**: BookingConfirmed 이벤트를 누가 소비하나요? 정산? 알림?
- **내 응답/판단**: 정산과 알림 둘 다. 이벤트로 팬아웃.
- **설계 변경 여부**: X
- **변경 내용**: 기존 설계 유지, 이벤트 소비자 목록만 명시.
