# 요약 노트 템플릿 (주차 정리용)

## 요구사항
- 링크/파일: `step1_requirements/requirement_card.md`
- 요약: 숙박 예약 상태/제약/정산 보류 규칙 정의.

## 설계 요약
- 예약관리(Booking), 체류라이프사이클(StayProgress), 정산관리(HostSettlement)로 경계를 나눔.
- Booking은 재고/상태 전이를, StayProgress는 체크인/체크아웃을, HostSettlement는 보류 정책을 책임진다.
- 이벤트(`BookingConfirmed`, `CheckInRecorded` 등)로 각 컨텍스트를 동기화한다.

## 핵심 결정 2개
1. 체크인/체크아웃은 StayProgress 애그리게잇에서만 허용하고 Booking은 상태/재고에 집중한다.
2. 정산 보류 여부는 HostSettlement가 이벤트/조회로 알려주며 Booking이 직접 판단하지 않는다.

## 버린 대안
- Booking 애그리게잇이 체크인/체크아웃까지 모두 수용: 상태 전이 검증이 복잡해지고 트랜잭션이 비대해져 포기했다.

## 시나리오 결과
- 정상 시나리오: 체크인 전 확정 (재고 OK → 확정됨)
- 예외/경계 시나리오: 정산 보류 거절, 체크인 이후 취소 시도, 체크아웃 순서 오류

## 테스트 결과
- 정상 케이스: 체크인 전 확정
- 예외/경계/위험 중 부족한 영역: 위험 케이스 후속 조치 (외부 파트너 알림) TODO

## 고민 포인트

- 정산 보류 정책 변화가 잦을 때 이벤트 순서를 어떻게 강제할까?

## 파괴적 요구사항 제안
- **요구사항**: "부분 취소 허용 - 3박 중 1박만 취소 가능"
- **왜 깨지나**: 현재 Booking은 예약 전체를 하나의 단위로 관리. 부분 취소하려면 예약 단위 분리 필요.
- **어디가 먼저 깨지나**: Booking 애그리게잇의 상태 전이 로직, 재고 점유 로직.
