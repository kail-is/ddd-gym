# 요구사항 카드

## 체크리스트
- [x] 목적이 사용자/비즈니스 문제로 표현되었다.
- [x] 주체/행동/규칙이 명확히 구분되었다.
- [x] 상태 변화와 실패 조건이 기술되었다.
- [x] 열린 질문이 최소 2개 있다.

## 포맷
- **목적**: 고객이 숙박 일정을 확정하고, 호스트는 체크인 준비 상태를 안전하게 관리한다.
- **주체**: 고객, 호스트, 예약 시스템, 정산 시스템.
- **행동**: 고객이 숙소/기간/인원을 선택해 예약 요청을 보내면, 호스트 혹은 자동 규칙이 승인을 결정한다.
- **규칙**:
  - 예약 상태는 `요청됨` → `확정됨`/`취소됨` → `체크인` → `체���아웃`을 따른다.
  - 이미 취소된 예약은 다시 확정할 수 없다.
  - 체크인 이후에는 취소할 수 없다.
  - 체크아웃 전에는 체크아웃 완료 상태로 전환할 수 없다.
- **상태 변화**: 요청 시 `요청됨`, 승인 시 `확정됨`, 고객 취소 시 `취소됨`, 입실 시 `체크인`, 퇴실 시 `체크아웃`.
- **실패 조건**:
  - 같은 숙소/기간 조합이 이미 `확정됨`.
  - 호스트 정산 보류 상태.
  - 숙소 유지보수로 예약 차단.
- **열린 질문**:
  1. 체크인 가능 여부를 누가 결정하며 어떤 데이터가 필요할까?
  2. 취소 사유와 패널티 관리는 어떤 컨텍스트가 담당해야 하나?
