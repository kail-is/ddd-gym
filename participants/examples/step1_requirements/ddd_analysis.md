# DDD 분석 템플릿

## 체크리스트
- [x] 컨텍스트/애그리게잇/이벤트 후보가 각각 1개 이상이다.
- [x] 규칙을 정책 vs 불변조건으로 분류했다.
- [x] 트랜잭션 경계를 설명했다.
- [x] 선택한 방식과 버린 대안을 비교했다.
- [x] 트레이드오프를 명시했다.

## 문서 구조

### 1. Bounded Context 후보
- **예약관리**: 숙소/날짜 재고 확인, 예약 요청 수락/거절, 취소 규칙의 1차 검증.
- **체류라이프사이클**: 체크인/체크아웃 가능 여부 판단, 현장 상태 기록.
- **정산관리**: 호스트 정산 보류, 지급 조건, 패널티 정리.

### 2. Aggregate 후보
- **Booking** (루트: Booking): 숙소/날짜/인원 조합을 점유하고 상태 전이를 관리.
- **StayProgress** (루트: StayProgress): 체크인/체크아웃, 객실 상태, 입실 제한을 추적.
- **HostSettlement** (루트: HostSettlement): 호스트 정산 보류 여부, 지급 가능 상태를 보관.

### 3. Domain Event 후보
- `BookingRequested`
- `BookingConfirmed`
- `BookingCancelled`
- `CheckInRecorded`
- `CheckOutRecorded`
- `HostSettlementBlocked`

### 4. 엔티티 / Value Object 후보
**엔티티**:
- Booking: 예약 식별자, 상태 전이 책임
- StayProgress: 체류 상태 추적, 체크인/아웃 기록

**Value Object**:
- StayPeriod: 체크인/체크아웃 날짜 범위
- GuestCount: 성인/어린이 인원 수
- BookingId: 예약 식별자

### 5. 규칙 분류
**불변조건**:
- `확정됨` 상태의 예약은 동일한 숙소/날짜 범위를 중복 점유할 수 없다.
- `체크인` 이전에는 `체크아웃` 상태로 전환할 수 없다.

**정책**:
- 체크인 24시간 이내 취소는 정책 위반으로 자동 거절한다.
- 정산 보류는 정산 컨텍스트에서만 변경하며 예약 컨텍스트는 결과를 조회만 한다.

### 6. 트랜잭션 경계
- 예약 생성 시 `재고 조회 + Booking 애그리게잇 생성 + 상태 요청됨→확정됨 결정`을 한 트랜잭션으로 처리.
- 체크인/체크아웃은 StayProgress 애그리게잇 단일 트랜잭션. 이후 정산/알림은 이벤트로 팬아웃.

### 7. 선택한 방식 vs 버린 대안
**선택**: 체크인/체크아웃을 StayProgress 애그리게잇으로 분리해 예약을 단순화.

**버린 대안**: Booking 애그리게잇이 체크인/체크아웃까지 모두 책임. 문제: 상태 전이 규칙이 혼재하고 트랜잭션이 비대해짐.

### 8. 트레이드오프
- 분리된 컨텍스트 간 이벤트 동기화 필요 → eventual consistency를 관리해야 한다.
- StayProgress 분리로 체크인 정책 실험은 쉬워지지만 조회 시 조인 비용이 늘어난다.
- 정산을 외부 컨텍스트에 위임하면 규칙 중복은 줄지만, 장애 시 예약 확정 로직이 영향을 받을 수 있다.

> TIP: AI에게는 "정답을 말하지 말고, 이 설계의 경계/책임 약점을 질문으로 드러내달라"고 요청하세요.
