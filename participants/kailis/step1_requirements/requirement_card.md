# 요구사항 카드
체크리스트를 모두 채우면 다음 단계(DDD 해석)로 진행합니다.

## 체크리스트
- [x] 목적이 사용자/비즈니스 문제로 표현되었다.
- [x] 주체/행동/규칙이 명확히 구분되었다.
- [x] 상태 변화와 실패 조건이 기술되었다.
- [x] 열린 질문이 최소 2개 있다.

## 포맷
- **목적**: 고객이 원하는 숙소/날짜를 예약 요청하고, 숙소가 이를 확정하여 체크인/체크아웃까지 흐름을 관리한다. 플랫폼은 둘을 연결하는 도구 역할.
- **주체**:
  - 고객: 예약 요청, 취소 요청, 체크인
  - 숙소(호스트): 확정, 거절, 취소 확정, 체크아웃
  - 플랫폼: 연결 도구 (주체 아님)
- **행동**:
  - 고객 → 예약 요청 (결제 포함)
  - 숙소 → 확정 / 거절
  - 고객 → 취소 요청 / 숙소 → 취소 확정 (또는 숙소가 직접 취소)
  - 고객 → 체크인
  - 숙소 → 체크아웃
- **규칙**:
  - 취소된 예약은 재확정 불가
  - 체크인 이후 취소 불가
  - 체크인 없이 체크아웃 불가
  - 취소 시 사유 기록 필수 (취소 주체, 사유)
- **상태 변화**:
  - `요청됨` → `확정됨` / `거절됨`
  - `확정됨` → `체크인` → `체크아웃`
  - `확정됨` → `취소됨` (사유 기록)
- **실패 조건**:
  - 거절됨 (숙소가 요청 거절) → 전액 환불
  - 취소됨 (고객/숙소 취소) → 조건부 환불
- **열린 질문**:
  1. 플랫폼이 가진 규칙의 범위는? (취소 수수료, 환불 정책 등)
  2. 환불 조건 상세는? (취소 시점, 취소 주체에 따른 차이)
  3. 숙소가 등록한 정보와 실제 상황이 다를 때 누가 책임지는가?

## 예시 (숙박 예약)
- **목적**: 고객이 원하는 숙소 날짜를 확정하고, 호스트는 체크인 전 준비 상태를 관리한다.
- **주체**: 고객, 호스트, 예약 시스템, 정산 시스템.
- **행동**: 고객이 숙소/날짜/인원을 선택해 예약 요청을 보내면 호스트 혹은 자동 규칙이 승인을 결정한다.
- **규칙**: 체크인 이후 취소 불가, 취소된 예약은 재확정 불가, 체크아웃 전 체크아웃 불가.
- **상태 변화**: `요청됨` → `확정됨`/`취소됨` → `체크인` → `체크아웃`.
- **실패 조건**: 날짜 중복, 호스트 정산 보류, 숙소 유지보수로 임시 차단.
- **열린 질문**: 체크인 승인 권한은 어디에 두나? 취소 사유는 어떤 컨텍스트가 저장하나?
