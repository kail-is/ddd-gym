## 정상 시나리오

### 예약 요청 → 결제 → 확정 (정상 플로우)
- **종류**: 정상
- **Given**: 숙소 A는 7/20~7/22 가용일이 남아있고, 다른 사용자의 잠금이 없다.
- **When**: 고객 B가 7/20~7/22로 예약을 요청하고 결제에 성공한다.
- **Then**: 예약 상태가 `요청됨` → `확정됨`이 되고, `BookingConfirmed` 이벤트가 발행된다. 해당 날짜는 이선좌 잠금에서 확정 잠금으로 전환된다.

### 체크인 → 체크아웃 (정상 플로우)
- **종류**: 정상
- **Given**: 예약이 `확정됨` 상태이고, 체크인 날짜가 도래했다.
- **When**: 고객이 체크인을 수행한다.
- **Then**: 예약 상태가 `체크인`이 되고 `CheckedIn` 이벤트가 발행된다.

### 고객 취소 요청 → 호스트 확정 → 환불 (정상 플로우)
- **종류**: 정상
- **Given**: 예약이 `확정됨` 상태이고, 체크인 전이다.
- **When**: 고객이 취소를 요청한다. 호스트가 취소를 확정한다.
- **Then**: CancellationRequest 생성 (Booking 상태 변경 없음) → 호스트 확정 시 `CancellationApproved` 발행 → Booking.cancel() 호출 → `BookingCancelled` 발행. 예약 컨텍스트가 환불 금액을 계산하여 결제 컨텍스트에 환불 요청. 숙소 컨텍스트가 가용일을 복구한다.

## 예외 시나리오

### 이선좌 잠금 중 다른 사용자 예약 시도 – 거절
- **종류**: 예외
- **Given**: 고객 B가 숙소 A의 7/20~7/22를 예약 요청하여 이선좌 잠금 중이다 (heartbeat 폴링 진행 중).
- **When**: 고객 C가 동일 숙소 동일 날짜로 예약을 요청한다.
- **Then**: 고객 C의 예약 요청이 거절된다. 사유: 해당 날짜 잠금 중.

### 결제 실패 후 재시도 성공
- **종류**: 예외 → 정상 복구
- **Given**: 고객 B가 예약 요청 후 결제를 시도했으나 실패했다. 예약 상태는 `요청됨` 유지, heartbeat 폴링 진행 중.
- **When**: 고객 B가 다른 결제 수단으로 재시도하여 성공한다.
- **Then**: 예약이 `확정됨`으로 전이, `BookingConfirmed` 발행.

### 결제 실패 + 폴링 끊김 → 잠금 해제
- **종류**: 예외
- **Given**: 고객 B가 예약 요청 후 결제에 실패했다. 이후 브라우저를 닫아 heartbeat 폴링이 중단되었다.
- **When**: 마지막 heartbeat로부터 10분이 경과한다.
- **Then**: 잠금이 해제되고 예약 요청이 삭제된다. `BookingExpired` 이벤트 발행 → 숙소 컨텍스트가 가용일을 복구한다. (예약 성립 전이므로 상태 전이가 아님)

### 취소된 예약 재확정 시도 – 거절
- **종류**: 예외
- **Given**: 예약이 `취소됨` 상태이다.
- **When**: 호스트가 해당 예약을 확정하려 한다.
- **Then**: 상태 전이 거부. "취소된 예약은 재확정 불가" 불변조건 위반.

### 체크인 없이 체크아웃 시도 – 거절
- **종류**: 예외
- **Given**: 예약이 `확정됨` 상태이다 (체크인 안 함).
- **When**: 숙소가 체크아웃을 처리하려 한다.
- **Then**: 상태 전이 거부. "체크인 없이 체크아웃 불가" 불변조건 위반.

### 취소 요청 대기 중 체크인 시도 – 거절
- **종류**: 예외
- **Given**: 예약이 `확정됨` 상태이고, 고객이 취소를 요청하여 CancellationRequest가 대기 중이다.
- **When**: 고객이 체크인을 시도한다.
- **Then**: 체크인 서비스가 CancellationRequest 큐를 확인, 대기 중인 요청이 있으므로 체크인 거절. (Booking 엔티티가 아닌 서비스 레벨에서 방어)

### 호스트가 취소 거절 – Booking 영향 없음
- **종류**: 예외
- **Given**: 예약이 `확정됨` 상태이고, 고객의 취소 요청(CancellationRequest)이 대기 중이다.
- **When**: 호스트가 취소를 거절한다.
- **Then**: `CancellationRejected` 발행. CancellationRequest 처리 완료. Booking 상태는 `확정됨` 그대로 유지. 고객은 체크인 가능해짐.

## (학습 목적) 부분 취소 시나리오

### 3박 중 1박 부분 취소
- **종류**: 예외 (현재 미지원, 학습용)
- **Given**: 고객 B가 7/20~7/23 (3박) 예약이 `확정됨` 상태이다. Booking 내부에 BookingItem 3개 (7/20, 7/21, 7/22).
- **When**: 고객이 7/21 날짜만 취소를 요청한다.
- **Then**: 7/21 BookingItem이 `취소됨`으로 전이. Booking 자체는 `확정됨` 유지. 예약 컨텍스트가 1박 분 환불 금액 계산 → 결제 컨텍스트에 부분 환불 요청. 숙소 컨텍스트가 7/21 가용일 복구.

### 전체 날짜 부분 취소 → Booking 취소
- **종류**: 예외 (현재 미지원, 학습용)
- **Given**: 고객 B가 7/20~7/22 (2박) 예약, BookingItem 2개. 7/20은 이미 취소됨.
- **When**: 고객이 7/21도 취소를 요청한다.
- **Then**: 전체 BookingItem이 `취소됨`. Aggregate Root가 자기 상태를 `취소됨`으로 전이. `BookingCancelled` 발행.
