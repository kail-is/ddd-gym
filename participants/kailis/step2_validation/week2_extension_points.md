> 결제 실패하면 예약은 어떻게 되나?

## 1. 결제 실패하면 예약은 어떻게 되나?

| 질문 | 은빈 |
|------|------|
| 결제 실패 시 예약 상태는? | `요청됨` 유지, heartbeat 폴링으로 잠금 유지 |
| 재시도 가능? 몇 분 내? | 폴링 중 무제한 재시도, 폴링 끊기면 10분 유예 |
| 재시도 중 다른 사람이 예약하면? | 이선좌 잠금 중이므로 거절 |
| 결제 실패 이벤트 발행? | PaymentFailed는 불필요, 만료 시 BookingExpired 발행 |
| 호스트에게 알림 가나? | 확정 전이라 불필요 |
| 내 설계에서 어디가 깨지나? | 잠금 상태/lastHeartbeat, BookingExpired 이벤트, 숙소 가용일 잠금 개념 추가 필요 |

### 상세 설명

**결제 실패 시 예약 상태 유지 전략: heartbeat 폴링**

결제 실패 시 별도 상태(`임시예약`, `결제대기` 등)를 추가하지 않고, `요청됨` 상태를 그대로 유지한다.
대신 클라이언트가 브라우저에서 주기적으로 heartbeat 폴링을 보내면서 "이 사용자가 아직 결제를 시도 중"임을 서버에 알린다.

- **이선좌(먼저 선점한 좌석) 방식**: 폴링이 살아있는 동안 해당 날짜는 잠금 상태. 다른 사용자가 같은 날짜로 예약 요청하면 거절.
- **잠금 해제 조건**: 폴링이 끊긴 시점으로부터 10분 경과. 서버는 `lastHeartbeat`를 기록하고, 마지막 heartbeat로부터 10분이 지나면 잠금 해제.
- **만료 시 처리**: 잠금 해제 시 `BookingExpired` 이벤트 발행 → 숙소 컨텍스트가 구독하여 가용일 복구.

**PaymentFailed 이벤트가 불필요한 이유**: 결제 재시도는 클라이언트와 결제 컨텍스트 안에서 끝나는 일이고, 다른 컨텍스트가 알아야 할 건 최종적으로 "예약이 만료됐다"는 사실뿐이다. 그래서 `PaymentFailed` 대신 `BookingExpired`로 충분하다.

**호스트 알림 불필요**: `요청됨` 상태는 아직 확정 전이므로 호스트에게 노이즈만 줄 뿐이다.

## 2. 부분 취소 가능하면 설계가 어디서 깨지나?

| 질문 | 은빈 |
|------|------|
| 3박 예약 중 1박만 취소하면? | 부분 취소 미지원, 전체 취소 → 재예약 (설계 결정). 단 공부 차원에서 부분 취소도 고민함 |
| Booking Aggregate 쪼개야 하나? | Booking은 1개 유지, 내부에 날짜별 BookingItem 추가 |
| 환불 계산은 누가? | 예약 컨텍스트가 계산, 결제 컨텍스트는 PG 통신만 |
| 취소된 날짜는 다시 예약 가능? | 가능, 숙소 컨텍스트에 가용일 복구 이벤트 발행 |
| 상태 전이가 복잡해지나? | Booking 상태는 유지, 내부 아이템별 취소. 전체 취소 시 Aggregate Root가 자기 상태 전이 |
| 내 설계에서 어디가 깨지나? | DateRange VO로는 부분 취소 표현 불가, 숙소 가용성도 날짜별 × 호수/수량별 관리로 변경 필요 |

### 상세 설명

**설계 결정: 부분 취소 미지원**

실제 서비스(에어비앤비 등)도 부분 취소를 지원하지 않는다. 전체 취소 후 원하는 날짜로 재예약하는 방식.
단, 재예약 사이에 다른 사용자가 해당 날짜를 선점할 수 있다는 리스크가 있다. 이 리스크를 감수하는 것이 설계 결정.

**부분 취소를 지원한다면 (학습 목적)**

`DateRange` VO는 연속된 날짜 구간을 하나로 표현하기 때문에, 중간 날짜만 빼는 것이 불가능하다.
이를 해결하려면 Booking 내부에 날짜별 `BookingItem`을 두어야 한다.

- **Booking (Aggregate Root)**: 전체 예약을 대표. 상태는 내부 아이템 상태로부터 도출.
- **BookingItem**: 날짜별 예약 단위. 개별적으로 취소 가능.
- **상태 전이**: 개별 아이템이 취소되더라도 Booking 자체는 `확정됨` 유지. 전체 아이템이 취소되면 Aggregate Root가 자기 상태를 `취소됨`으로 전이. 이건 파생 상태 동기화 문제가 아니라, Aggregate Root가 자기 불변조건을 지키는 정상적인 패턴.

**환불 계산 책임: 예약 컨텍스트**

결제 컨텍스트는 BFF성으로 PG 통신만 담당한다. 환불 금액 계산 같은 비즈니스 로직은 예약 컨텍스트가 담당하고, 결제 컨텍스트에는 "이 금액 환불해줘"만 전달한다. PG사 차이(토스페이먼츠 부분환불 지원 등)는 결제 컨텍스트 안에서 흡수.

**숙소 가용성 관리 변경**

부분 취소가 들어오면 숙소 컨텍스트의 가용성 관리도 단순 DateRange가 아닌, 날짜별 × 호수(또는 수량)별 관리로 변경이 필요하다.

## 3. 설계가 깨진 부분

| 질문 | 은빈 |
|------|------|
| 어디가 깨졌나? | DateRange VO, 숙소 가용성 관리 |
| 왜 깨졌나? | 부분 취소 시 연속 날짜 가정이 무너짐 |
| Aggregate 경계 문제? | 환불 금액 계산 책임이 예약 컨텍스트로 명확해짐, 결제는 BFF성 PG 통신만 |
| 이벤트 순서 문제? | BookingExpired 추가, 부분 취소 시 가용일 복구 이벤트 필요 |
| 상태 전이 문제? | 없음. 요청됨 유지 + heartbeat 폴링으로 해결 |

### 상세 설명

**깨진 핵심: DateRange VO의 연속 날짜 가정**

Week1 설계에서 `DateRange`는 체크인~체크아웃을 하나의 연속 구간으로 표현한다. 부분 취소가 들어오면 7/20~7/23 중 7/21만 빼야 하는데, 이 구조로는 표현 불가.

**Aggregate 경계가 명확해진 부분**

결제 실패/부분 취소 고민을 통해 예약과 결제 컨텍스트의 책임이 더 명확해졌다.
- 예약 컨텍스트: 상태 관리, 잠금, 환불 금액 계산
- 결제 컨텍스트: PG 통신 (결제/환불 실행)

**새로 추가된 이벤트**

- `BookingExpired`: 결제 실패로 인한 예약 만료 시 발행. 숙소 컨텍스트가 구독하여 가용일 복구.
- 부분 취소 시에도 동일하게 가용일 복구 이벤트 필요.

## 4. 해결 방안

| 질문 | 은빈 |
|------|------|
| 해결 방안 | BookingItem 추가 (부분 취소), heartbeat 폴링 + BookingExpired (결제 실패) |
| 트레이드오프 | 폴링은 서버 TTL로도 충분하지만 가용일 빠른 해제로 예약 기회 증가. 부분 취소는 공부 목적, 실제론 전체 취소로 충분 |
| 다른 곳에 영향? | 기존 설계 구조 변경 없음, 확장성 옵션 추가 수준 |

### 상세 설명

**해결 방안 1: heartbeat 폴링 + BookingExpired (결제 실패 대응)**

서버 TTL만으로도 동작하지만, heartbeat 폴링을 도입하면 사용자가 실제로 이탈한 시점을 더 빠르게 감지할 수 있다. 이로 인해 다른 사용자가 해당 숙소를 더 빨리 예약할 수 있게 되어 전체 예약 기회가 증가한다.

**해결 방안 2: BookingItem 추가 (부분 취소 대응)**

실제 서비스에서는 부분 취소를 지원하지 않기로 결정했다. 전체 취소 → 재예약 방식이 충분하다.
다만 학습 목적으로 부분 취소를 설계해 본 결과, `DateRange` → 날짜별 `BookingItem`으로의 분해가 필요하며, 숙소 가용성 관리도 날짜별로 세분화되어야 한다는 점을 확인했다.

**트레이드오프 정리**

두 해결 방안 모두 기존 설계의 뼈대(Bounded Context 분리, 이벤트 기반 연결, Aggregate 구조)를 바꾸지 않는다. 확장성 옵션이 추가된 수준이며, 필요에 따라 채택 여부를 결정할 수 있다.

## 추가 설계 토론: 취소 요청 흐름

### 설계 결정

취소에도 확정/거절 흐름을 도입했다. 고객이 취소를 요청하면 호스트가 확정 또는 거절한다.

### 핵심 고민: CancellationRequest는 어디에 두나?

**선택지 검토:**

1. **Booking 엔티티 내부 필드** — `pendingCancellationRequest` 같은 필드 추가
   - 장점: 불변조건(취소 요청 중 체크인 불가)을 엔티티 `require`로 보호 가능
   - 단점: Booking에 관심사가 섞임. 온보딩 시 "이 필드 뭐야?" 설명 비용

2. **별도 Aggregate** — 독립적으로 조회/저장
   - 기각: 내부에 보호할 불변조건이나 하위 객체가 없음. Aggregate는 과함

3. **Booking 밖 독립 엔티티 (큐 테이블)** — 히스토리성 테이블로 관리 ← **채택**
   - Booking은 취소 요청의 존재를 모름
   - 호스트가 대기 중인 요청을 API로 조회
   - 확정 시 `CancellationApproved` 이벤트 발행 → Booking.cancel() 호출
   - 거절 시 요청 처리 완료, Booking은 아무 영향 없음

### 핵심 고민: 취소 요청 시 Booking 상태를 바꿔야 하나?

**결론: 안 바꾼다.**

취소가 확정됐을 때만 `확정됨 → 취소됨`으로 전이. 요청 단계에서 상태를 바꾸면 거절 시 되돌려야 하는 복잡도가 생김.

### 핵심 고민: 취소 요청 중 체크인 방지

Booking 상태를 안 바꾸면 `확정됨`인 상태에서 체크인이 가능해지는 문제가 있다.

**선택지:**
1. Booking에 `취소요청됨` 상태 추가 → 체크인을 엔티티 레벨에서 막음
2. 서비스 레벨에서 CancellationRequest 큐를 확인하여 체크인 거절 ← **채택**

**채택 이유:**
- Booking 엔티티에 취소 요청 필드를 넣으면 관심사가 섞이고, 새로운 사람에게 "이 필드 뭔데?" 가르쳐야 함
- 엔티티에 넣어도 휴먼에러(필드 안 넣고 호출) 확률은 서비스 우회 확률과 비슷
- 보완: Kotlin `internal` 가시성으로 서비스 외부에서 직접 호출 제한 가능
