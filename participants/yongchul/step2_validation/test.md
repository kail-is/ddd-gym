## 테스트 케이스

| 분류 | 케이스명 | 조건 | 기대 결과 | 실패 조건 | 참고 시나리오 |
|------|----------|------|-----------|-----------|---------------|
| 정상 | 예약 요청 → 결제 완료 → 확정됨 | Redis 선점 키 없음, 결제 성공 | Booking 상태 `확정됨`, `BookingConfirmed` 발행 | 상태가 `요청됨` 유지되거나 이벤트 미발행 | 시나리오 1 |
| 정상 | 체크인 처리 | Booking `확정됨`, 체크인 날짜 도래 | Booking 상태 `체크인됨`, `CheckInRecorded` 발행 | 상태 미변경, 이벤트 미발행 | 시나리오 2 |
| 정상 | 체크아웃 처리 | Booking `체크인됨`, 체크아웃 날짜 도래 | Booking 상태 `체크아웃됨`, `CheckOutRecorded` 발행 | 상태 미변경, 이벤트 미발행 | 시나리오 2 |
| 예외 | TTL 만료 후 결제 완료 | 선점 TTL 만료됨, Booking `만료됨` 상태 후 `TransactionCompleted` 수신 | Booking 상태 `만료됨` 유지, 결제 취소 처리, 오류 메시지 반환 | Booking이 `확정됨`으로 전이되거나 결제가 취소되지 않음 | 시나리오 3 |
| 예외 | 체크인 이후 취소 시도 | Booking `체크인됨` 상태 | 취소 거절, 불변조건 위반 오류, 상태 유지 | 취소 처리됨 | 시나리오 4 |
| 예외 | 체크인 없이 체크아웃 시도 | Booking `확정됨`, 체크인 이력 없음 | 체크아웃 거절, 불변조건 위반 오류, 상태 유지 | 체크아웃 처리됨 | 시나리오 5 |
| 경계 | 취소된 예약 재확정 시도 | Booking `취소됨` 상태 | 재확정 거절, 불변조건 위반 오류, 상태 `취소됨` 유지 | 재확정 처리됨 | 시나리오 6 |
| 경계 | 중복 예약 시도 (선점 키 존재) | 동일 숙소·동일 일정에 Redis 선점 키 이미 존재 | 예약 거절, "이미 선점 중인 일정" 오류 | 선점 성공하여 더블 부킹 발생 | 시나리오 1 (역케이스) |
| 위험 | TransactionCompleted 중복 수신 | Booking 이미 `확정됨`, `BookingConfirmed` 처리 완료 | 중복 이벤트 무시, 상태 변경 없음 | 중복 확정 처리로 일정 데이터 오염 | 시나리오 7 |
| 위험 | SchedulePreemptionExpired 와 TransactionCompleted 순서 역전 방지 | 두 이벤트 모두 Kafka key = BookingID로 발행 | 동일 파티션 내 순서 보장, `만료됨` 처리 후 결제 취소 | 순서 역전으로 결제가 정상 확정됨 | 시나리오 3 |
| 정상 | 결제 실패 후 TTL 내 재시도 성공 | Booking `REQUESTED`, 선점 키 유효, 신규 Transaction 생성 후 결제 성공 | Booking `CONFIRMED`, 선점 키 삭제, 실패 Transaction `FAILED` 이력 보존 | 선점 키가 삭제되어 재시도 불가하거나 기존 Transaction 재사용 | 시나리오 8 |
| 예외 | 결제 실패 후 TTL 만료 | Booking `REQUESTED`, 결제 실패 상태에서 TTL 만료 | Booking `EXPIRED`, 미완료 Transaction `FAILED` 유지, 결제 취소 불필요 | Booking이 `CONFIRMED`로 전이되거나 불필요한 결제 취소 발생 | 시나리오 9 |
| 경계 | 결제 실패 직후 선점 키 유지 확인 | `TransactionFailed` 발행 직후 Redis 키 조회 | 선점 키 존재, TTL 잔여 시간 유지 | 키 삭제됨 | 시나리오 8, 9 |
