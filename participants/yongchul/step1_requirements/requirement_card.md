## 목적

- **고객 관점**: 원하는 숙소의 특정 일정을 선점하고, 결제까지 완료하여 숙박을 확정한다.
- **호스트 관점**: 자신의 숙소 일정을 관리하고, 확정된 예약에 대해 체크인/체크아웃을 처리한다.
- **비즈니스 핵심 문제**: 동일 숙소·동일 일정에 대한 더블 부킹(이중 예약)을 방지하면서, 고객의 예약 경험을 보장한다.

---

## 주체

| 주체 | 역할 |
|------|------|
| 고객 | 숙소 검색, 일정 선택, 예약 요청, 결제 |
| 호스트 | 숙소 정보 등록, 일정 관리, 체크인/체크아웃 처리 |
| 예약 시스템 | 예약자 정보 및 예약 데이터 관리 |
| 숙소 시스템 | 숙소 정보, 일정 선점/확정/충돌 방지 관리 |
| 결제 시스템 | 결제 금액 계산, 결제 요청/완료/실패 처리 |

---

## 행동

1. 고객이 숙소와 일정을 선택한다.
2. 고객이 "예약하기"를 누르는 시점에 숙소 도메인이 Redis 키(숙소ID + 일정)를 생성하여 일정을 선점한다.
3. 고객이 예약자 정보를 입력하고 결제 금액을 확인한다.
4. 고객이 결제를 요청한다.
5. 결제가 완료되면 Kafka 이벤트를 통해 숙소 도메인에 확정 신호를 전달한다.
6. 숙소 도메인이 해당 일정을 "예약 완료" 상태로 생성/변경하고 Redis 키를 삭제한다.
7. 결제가 실패하면 결제 도메인이 숙소 도메인에 삭제 요청을 보내고, Redis 키가 제거된다.

---

## 규칙

### 불변조건 (절대 깨지면 안 됨)
- 동일 숙소·동일 일정에 두 개의 확정된 예약이 존재할 수 없다. (더블 부킹 불가)
- 취소된 예약은 재확정할 수 없다.
- 체크인 이후 취소할 수 없다.
- 체크인 없이 체크아웃할 수 없다.

### 정책 (변경 가능한 운영 규칙)
- 일정 선점 TTL은 기본 1시간이며, 숙소별/이벤트 기간별로 조절 가능하다.
- TTL 만료 시 선점이 자동 해제되어 다른 고객이 예약 가능해진다.
- 결제 실패 시 숙소 도메인에 명시적 삭제 요청을 보낸다.

---

## 상태 변화

### 예약 상태
```
요청됨 → 확정됨 / 취소됨 → 체크인 → 체크아웃
```

### 숙소 일정 상태
```
선택가능 → 선점됨(Redis 키 생성) → 예약완료(결제 확정)
                  ↓ (TTL 만료 or 결제 실패)
              선택가능
```

---

## 실패 조건

| 실패 상황 | 처리 |
|-----------|------|
| 선점 시 이미 Redis 키 존재 | "이미 예약 중인 일정입니다" 안내, 예약 불가 |
| 결제 시간 초과 (TTL 만료) | 선점 자동 해제, 다른 고객 예약 가능 |
| 결제 실패 | 결제 도메인 → 숙소 도메인 삭제 요청, 선점 해제 |
| 취소된 예약 재확정 시도 | 숙소 도메인에서 차단 |
| 체크인 후 취소 시도 | 예약 도메인에서 차단 |

---

## 열린 질문

1. TTL 만료로 선점이 해제되는 순간, 결제가 완료된 Kafka 이벤트가 도착하면 숙소 도메인은 어떻게 처리해야 하는가? (타임아웃 vs 결제완료 이벤트의 선순위)
2. 여러 외부 채널(자사 앱, OTA, 전화 예약)에서 동시에 예약이 인입될 때, 숙소 도메인의 충돌 방지 계층이 단일 진실 소스 역할을 하는가?
3. 숙소별 TTL 정책은 어디에 저장하고, 선점 시 어떻게 조회하는가?
4. 호스트가 직접 일정을 차단(유지보수 등)하는 경우, 이미 선점 중인 고객의 예약은 어떻게 처리되는가?
