# 질문 루프 로그

## 체크리스트

- [x] 설계에 영향을 준 질문 표시됨.

### 질문 1
- **영역**: 이벤트
- **질문 내용**: `RoomReserved`는 `BookingConfirmed`의 결과인가요, 아니면 `BookingConfirmed`가 되기 위한 전제인가요?
- **내 응답/판단**: `RoomReserved`는 전제다. 동일 기간 점유가 먼저 성공하고, 모든 확정 조건이 충족된 뒤 `BookingConfirmed`가 발생한다.
- **설계 변경 여부**: O
- **변경 내용**: `RoomReserved`를 예약 확정 전제 사실로 정리하고, `BookingConfirmed`는 최종 확정 사실로 의미를 구분했다.

### 질문 2
- **영역**: 상태
- **질문 내용**: `BookingCancelled`는 하나의 의미만 가지나요, 아니면 서로 다른 실패/취소 상황을 함께 담고 있나요?
- **내 응답/판단**: 하나의 이벤트 안에 여러 `reason`을 담는다. `USER_CANCELED`, `PAYMENT_FAILED`, `HOLD_EXPIRED` 등 종료 이유를 함께 표현한다. 종료 이유를 모두 별도 상태로 분리하면 오히려 복잡도가 증가하므로, `Booking`이 최종 상태만 가진다는 전제를 유지한다.
- **설계 변경 여부**: X
- **변경 내용**: 이벤트 이름은 유지하고, 종료 원인은 `reason`으로 구분하는 방향을 유지했다.

### 질문 3
- **영역**: 트랜잭션
- **질문 내용**: 현재 트랜잭션 경계가 "예약 확정 시점"이라는 한 문장만으로 실패 시 복구 책임까지 설명되나요?
- **내 응답/판단**: 부족하다. Booking 상태 전이는 하나의 트랜잭션으로 처리하지만, `RoomAvailability`와 `Payment`는 별도 트랜잭션이며 실패 시 보상 주체를 명시해야 한다.
- **설계 변경 여부**: O
- **변경 내용**: `ddd_analysis.md`의 트랜잭션 경계를 Application Service/Saga 중심 보상 흐름까지 설명하도록 보강했다.

### 질문 4
- **영역**: 책임
- **질문 내용**: `REQUESTED` 안에 "아직 시도 안 함"과 "중간에 실패함"이 함께 들어가면, 재시도 가능 여부를 누가 구분하나요?
- **내 응답/판단**: `Booking`의 `REQUESTED`는 "최종 확정 전"이라는 업무 상태만 뜻한다. 결제 재시도 가능 여부는 `Payment`의 `retryCount`와 `status`, 예약 유지 가능 여부는 `RoomAvailability`의 `HOLD` 정책이 판단한다.
- **설계 변경 여부**: X
- **변경 내용**: `REQUESTED`를 진행 중 상태로 유지하되, 재시도 책임은 `Payment`, 유지 책임은 `HOLD` 정책에 둔다.

### 질문 5
- **영역**: 이벤트
- **질문 내용**: 보상 시작 신호는 `BookingConfirmed`의 부재인가요, 아니면 `RoomReserved` 후 실패라는 별도 사실인가요?
- **내 응답/판단**: `BookingConfirmed`의 부재가 아니라, `RoomReserved` 후 실패라는 별도 사실과 `BookingCancelled(reason=BOOKING_STATE_TRANSITION_FAILED)` 같은 명시적 신호가 필요하다.
- **설계 변경 여부**: O
- **변경 내용**: 보상 시작을 "이벤트가 안 왔다"가 아니라 "실패 사실이 발생했다"는 신호 중심으로 설명하도록 정리했다.

### 질문 6
- **영역**: 상태
- **질문 내용**: `REQUESTED` 상태에서는 사용자가 어떤 행동을 할 수 있나요?
- **내 응답/판단**: 사용자는 취소하는 것 외에는 동작할 수 없다. 결제 재시도와 `HOLD` 연장 같은 내부 진행은 시스템이 처리한다.
- **설계 변경 여부**: X
- **변경 내용**: `REQUESTED` 중 사용자 동작은 제한되고 내부 진행만 이어진다는 해석을 유지했다.

### 질문 7
- **영역**: 경계
- **질문 내용**: `RoomReserved` 후 실패라는 사실은 도메인에서 중요한 상태 변화인가요, 아니면 보상 오케스트레이션을 위한 운영 신호인가요?
- **내 응답/판단**: 사용자가 이해하는 상태가 아니라, 확정 진행 중 일부 단계가 실패했다는 운영 신호다.
- **설계 변경 여부**: O
- **변경 내용**: 이 사실을 사용자 상태가 아니라 보상 흐름의 시작점으로 해석했다.

### 질문 8
- **영역**: 상태
- **질문 내용**: 지금 가장 먼저 다시 써야 하는 문장은 무엇인가요?
- **내 응답/판단**: `Booking.status`는 "최종 결과 상태"만 표현한다.
- **설계 변경 여부**: O
- **변경 내용**: `REQUESTED`를 포함한 `Booking.status` 설명을 진행 세부 단계가 아닌 최종 결과 중심으로 다시 해석했다.

### 질문 9
- **영역**: 상태
- **질문 내용**: `REQUESTED`의 진행 중 의미를 조회 모델 없이 유지할 수 있나요?
- **내 응답/판단**: 가능은 하지만 해석 비용이 크다. `REQUESTED`가 확정 시도 전, 결제 재시도 중, `HOLD` 연장 중, 보상 대기 중을 함께 품기 때문에 조회 모델이 있으면 더 낫다.
- **설계 변경 여부**: O
- **변경 내용**: 진행 중 의미를 상태값이 아닌 조회 모델/운영 관점에서 보강하는 방향을 고민 포인트로 남겼다.
