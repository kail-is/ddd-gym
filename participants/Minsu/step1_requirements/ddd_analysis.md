# DDD 분석 템플릿
요구사항을 해석해 설계 후보를 만들고, 선택/트레이드오프를 명시합니다.

## 체크리스트
- [X] 컨텍스트/애그리게잇/이벤트 후보가 각각 1개 이상이다.
- [X] 규칙을 정책 vs 불변조건으로 분류했다.
- [X] 트랜잭션 경계를 설명했다.
- [X] 선택한 방식과 버린 대안을 비교했다.
- [X] 트레이드오프를 명시했다.

## 문서 구조
1. **Bounded Context 후보**
   - 예약 관리 - 예약 생성/확정/취소/체크인/체크아웃
   - 숙소 관리 - 숙소 정보, 가격 정책
   - 정산 관리 - 예약/취소 금액 기록
   - 결제 관리 - 결제 처리
2. **Aggregate 후보**
   - Reservation (루트: Reservation, 예약 금액 스냅샷 포함)
   - PricingPolicy (루트: Room, 호스트가 숙소에 설정하는 가격 정책)
3. **Domain Event 후보**
   - ReservationRequested - 예약 요청됨
   - ReservationConfirmed - 예약 확정
   - CustomerReservationCancelled - 사용자 예약 취소
   - HostReservationCancelled - 호스트 예약 취소
   - ReservationCheckedIn - 체크인
   - ReservationCheckedOut - 체크아웃
4. **Domain Service 후보**
   - ReservationOverlapChecker: 중복 확인
     - 입력: 숙소ID, 날짜, 시간
     - 역할: 해당 시간대 중복 예약 존재 여부 확인
5. **엔티티 / Value Object 후보**
   - 엔티티: Reservation (reservationId)
   - VO: 숙소ID(참조), 고객ID(참조), 시간대, 인원, 금액 스냅샷, 상태
6. **규칙 분류**
   - 불변조건
     - 예약 정보 변경 불가
     - 취소된 예약은 재확정 불가
     - 체크인 이후 취소 불가
     - 체크인 전 체크아웃 불가
     - 요청됨 상태의 시간대는 재요청 불가
   - 정책
     - 고객 취소 시 체크인 날짜 기준 비용 발생(비용이 고정이지 않음)
     - 호스트 취소 시 예약일 기준 보상 발생(비용이 고정이지 않음)
7. **트랜잭션 경계**
   - 단일 트랜잭션:
     - 예약 생성: 중복 확인 + 예약 관리 업데이트
     - 취소: 예약 정보 확인 + 예약 관리 업데이트
     - 체크인/체크아웃: 예약 정보 확인 + 예약 관리 업데이트
   - 이벤트 기반 처리 (별도 컨텍스트):
     - 예약 확정/취소 → 이벤트 발행 → 정산 관리 업데이트
8. **선택한 방식 vs 버린 대안**
   - 선택: 예약 관리와 정산 관리를 별도 컨텍스트로 분리                                                                                                                  
   - 버린 대안: 단일 컨텍스트에서 정산까지 함께 처리
9. **트레이드오프**
   - 장점: 각 시스템 변경이 다른 시스템에 영향 최소화, 독립 배포 가능
   - 단점: 시스템 간 단절 시 데이터 일관성 처리 복잡도 증가
    
> TIP: AI에게는 "정답을 말하지 말고, 이 설계의 경계/책임 약점을 질문으로 드러내달라"고 요청하세요.
