# 요구사항 카드
체크리스트를 모두 채우면 다음 단계(DDD 해석)로 진행합니다.

## 체크리스트
- [X] 목적이 사용자/비즈니스 문제로 표현되었다.
- [X] 주체/행동/규칙이 명확히 구분되었다.
- [X] 상태 변화와 실패 조건이 기술되었다.
- [X] 열린 질문이 최소 2개 있다.

## 포맷
- **목적**: 
  - 고객 선택의 시간대별 숙소 제공 앱
- **주체**: 
  - 고객, 호스트, 예약 시스템, 결제 시스템, 정산 시스템, 스케쥴링 시스템
- **행동**:
  - 고객 숙소&&날짜&&시간&&인원 선택하여 예약
  - 호스트: 숙소&&날짜&&시간&&인원에 따른 가격 설정 및 예약 가능여부 설정
- **규칙**: 
  - 요청됨 이후 3시간이 지나야 확정됨으로 변경
  - 요청됨 상태의 시간때는 재 요청할수 없다.
  - 예약 정보 변경 불가
  - 취소된 예약은 다시 확정 불가
  - 체크인 이후 취소 불가
  - 체크인 전 체크아웃 불가
- **상태 변화**:
  - 요청됨 => 확정됨
  - 요청됨 => 취소됨
  - 요청됨 => 확정됨 => 취소됨
  - 요청됨 => 확정됨 => 체크인 => 체크아웃
- **실패 조건**:
  - 네트워크 오류
  - 동일한 객실을 이미 다른사용자가 예약한경우
  - 예약 시간이 이미 지나간 경우
- **열린 질문**:
  - 예약 시스템이 3시간 후 자동 확정을 처리한다면,                                                            
    그 3시간 동안 시스템 장애가 발생했을 때 상태 전환은 어떻게 보장하는가?
  - 스케줄링 시스템 장애 시 "요청됨" 상태가 무기한 유지된다면,                                                
    해당 시간대는 영구적으로 점유되는가? 이 리스크를 어떻게 다룰 것인가?
  - 예약 확정/취소 시 정산 시스템과의 일관성을                                                                                                                          
    이벤트 기반으로 보장한다면, 이벤트 발행이 실패하거나                                                                                                                
    정산 시스템이 이벤트를 처리하지 못했을 때는 어떻게 되는가?
