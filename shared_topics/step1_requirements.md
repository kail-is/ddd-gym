# Week 1 – 요구사항/용어/컨텍스트

## 공통 목표
- 요구사항 템플릿 완성과 용어 정의.
- 컨텍스트 후보 2개 이상 도출.

## 도메인 브리핑
- 모두 **숙박 예약** 도메인을 다룹니다. 설계-공유-토의 흐름을 위해 같은 요구사항을 시작점으로 사용하세요.
- 예약 상태 플로우: `요청됨` → `확정됨`/`취소됨` → `체크인` → `체크아웃`.
- 제약: 취소된 예약은 다시 확정할 수 없고, 체크인 이후 취소 불가, 체크아웃 전 체크아웃 불가.
- **락(분산락 등) 설명은 금지**이며, 경계/책임으로 규칙을 지켜야 합니다.

## 설계 질문 힌트
- 취소 사유, 체크인 가능 여부, 재확정 차단은 각각 어디서 책임지나요?
- 규칙이 두 위치에 생기면 어떤 리스크가 생기나요?
- 테스트를 한 줄로 쓴다면 무엇을 호출해야 하나요?

## 토론 질문
- 유비쿼터스 언어로 표현되지 않은 용어는?
- 운영 정책과 불변조건을 어떻게 구분했나?
- 컨텍스트 경계 기준을 무엇으로 잡았나?

## 제출 체크리스트
- [ ] `01_templates/step1_requirement_card.md` 기반 문서.
- [ ] 용어 정의 목록 (10개 내외).
- [ ] 컨텍스트 다이어그램 스케치 혹은 서술.

## 오프라인 포인트
- 같은 요구사항을 서로 다른 경계로 나눈 사례 비교.
- \"예약 하나를 일부러 망가뜨리며 다시 설계\" 흐름을 공유하고, 사고 방식이 바뀐 순간을 포착해봅니다.
